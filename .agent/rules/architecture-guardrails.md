# 架构守护规则（原则优先）

> 配合 `/tech-debt-refactor`。**本规则 > 项目文档**；文档与原则冲突时，修订文档或提 ADR，而非堆兼容代码。

## 优先级

1. 代码 + 测试所证明的行为
2. 本文件原则（P1–P8，见 command §0）
3. 经审计的 ADR / 域内文档
4. 入口速查（CLAUDE.md 等）— 仅作索引，非细则 SSOT

## 写代码前

- 明确模块边界与对外契约
- 找到现有扩展点；若无且预期会增长，先设计扩展点再写功能
- 搜索是否已有相同语义实现（避免第二份 Map/枚举/handler）

## 禁止

- 为对齐过时文档而保留冗余分支或双入口
- 无截止日的「兼容旧版」路径
- 在 UI/Controller 硬编码本应由 catalog/config/i18n 提供的展示或策略
- 平行 `*Service2`、纯搬迁的 Helper/Util
- catch 吞错换默认值（无 log / 无测试）
- 单会话大范围重构（>300 行 diff 先拆任务）
- 新增文档重复已有规则（应合并或链接）

## 必须

- 上新则删旧（同 PR 或紧接 PR）
- 改行为则改/补测试或说明验证步骤
- 非显然逻辑：中文注释解释**为什么**
- 改架构：更新或新增简短 ADR；入口文档只改链接与一句摘要
- 发现文档债：记入 `docs/tech-debt-register.md`，不假装文档永远正确

## AI 会话纪律

- 新增行数 > 删除行数 ×2 → 先列可删项
- 不得引用未读/未与代码交叉验证的文档作为不可改依据
- 跨模块问题只登记，不顺手改
