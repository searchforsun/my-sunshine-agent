# 阶段 1.5 会话 MVP — 技术设计

> 日期：2026-06-11 | 状态：待实施  
> 对标：[implementation-plan.md](../../implementation-plan.md)  
> 前置：阶段一核心链路可跑通（BFF → Orchestrator → LLM Gateway SSE）  
> **扩展**：Track G（阶段 1.6）依赖 Track A～F 落地后实施

## 背景

前端已具备多会话 UI（`chatStore` + localStorage），但后端无持久化、无多轮上下文、无按用户隔离。BFF/Orchestrator 模型中 `conversationId` 字段已预留但未使用；Orchestrator 使用**全局单例** `InMemoryMemory`，所有用户/会话共享且重启丢失。

阶段 1.5 在**不引入 Sa-Token** 的前提下，补齐产品 MVP 所需的四项能力：MySQL 持久化、多轮上下文、轻量用户隔离、**流式断点续传**（刷新/断线/Stop 后可继续生成）。

## 已锁定决策

| # | 决策项 | 结论 |
|---|--------|------|
| 1 | Stop 时 assistant 部分响应 | **持久化 partial**（`doOnCancel` / `doFinally` 写入已累积内容） |
| 2 | history 上限 | **固定 20 条**，配置项 `agent.history.max-messages` 预留，MVP 默认 20 |
| 3 | 数据面落点 | **Orchestrator 统一**（CRUD + 写库）；BFF **纯转发** |
| 4 | 断点续传语义 | **同一条 assistant 消息续写**（不新增 user 轮次）；状态机驱动 |
| 5 | 续传触发方式 | **默认手动**（展示「继续生成」按钮）；`agent.resume.auto-on-reload` 默认 `false` |
| 6 | 流式中途落库 | **周期性 flush partial**（默认 500ms）+ 终态一次 commit |
| 7 | 续传 LLM 策略 | **Assistant 前缀续写**（history + 未完成 assistant partial → 继续 stream）；首版仅 **simple 意图** |
| 8 | 前端重叠去重 | 续传首包可能与 partial 前缀重叠，客户端 **overlap trim** 后再 append |
| 9 | 无感重连 vs 消息续写 | **分层**：Track G 负责同一次 generation 断线追读；Track F 负责 generation 已结束后的 LLM 续写 |
| 10 | Redis 缓冲位置 | **Orchestrator** 写 Redis Stream；BFF 只转发 `generationId` + `afterSeq` |
| 11 | 生成与推送解耦 | 客户端 SSE 断开 **不 cancel** LLM Job（在 orphan 超时前） |
| 12 | 重连参数 | 自定义 **`afterSeq`**（fetch POST SSE 无标准 Last-Event-ID）；SSE `id:` 字段单调递增 |

| 轨道 | 模块 | 缺口 | 改动性质 |
|------|------|------|---------|
| A | orchestrator | MySQL 持久化 | pom + Flyway + Entity + Service |

**`chat_message` 状态机（Track F）：**

```
streaming ──complete──► completed
    │
    ├──cancel/stop/disconnect──► interrupted  ──resume──► streaming ──► completed
    │
    └──error──────────────────► failed        ──resume──► streaming ──► completed|failed
```

| status | 含义 | 可续传 |
|--------|------|:------:|
| `streaming` | 生成中（含服务端 orphan：客户端已断开但服务端未感知） | 视 orphan 超时策略 |
| `interrupted` | 用户 Stop / 客户端断线 / 刷新，已有 partial | ✅ |
| `failed` | LLM 或内部错误，可能有 partial | ✅（MVP 同 interrupted） |
| `completed` | 正常结束 | ❌ |

**orphan `streaming` 处理：** 超过 `agent.generation.orphan-timeout`（默认 60s）无 chunk 写入，后台 job 或下次 resume 请求将其标记为 `interrupted`（content 保持已有 partial）。

**id 生成：** `UUID.randomUUID().toString().replace("-", "")`（64 字符内）

## Track B — 多轮上下文 + 流式写库（Orchestrator）

### 问题

1. `ChatController` 忽略 `conversationId`，每次只处理单条 `content`
2. `LlmGatewayClient.streamDirectly(String)` 仅发 1 条 user message
3. `SunshineAgent.chat(String, ...)` 仅 `List.of(singleMsg)`，依赖全局 Memory
4. 流式结束后无 assistant 消息落库
5. 刷新/断线后无法从断点继续，只能看到 partial（若已 flush）或丢失

### 设计

**chat/stream 新流程（新消息）：**

```
1. resolveConversation(conversationId?, userId, tenantId)
   - null/空 → ConversationService.create()
   - 有值   → getOwned() 校验归属
2. appendMessage(convId, USER, content, completed)
3. assistantMsg = appendMessage(convId, ASSISTANT, "", streaming)  # 预占位
4. history = loadHistory(convId, maxMessages)  # 不含当前 streaming assistant
5. 意图分流 → simple / knowledge；intent 写入 assistantMsg
6. 流式输出 SSE（见下方 meta/chunk 事件）
7. 累积 assistant 全文到 StringBuilder；每 flush-interval-ms  updateMessageContent(partial, streaming)
8. doOnComplete → updateMessageContent(full, completed)
   doOnCancel  → updateMessageContent(partial, interrupted)
   doOnError   → updateMessageContent(partial, failed)
9. 首条 user 消息且 title='新对话' → updateTitle(截取 content 前 28 字)
```

**chat/stream 续传流程（Track F）：**

请求体扩展（`content` 与 `resumeMessageId` **互斥**）：

```json
{ "conversationId": "<id>", "resumeMessageId": "<assistantMsgId>" }
```

```
1. getOwned(conversationId) + getMessageOwned(resumeMessageId)
2. 校验：role=assistant，status ∈ {interrupted, failed, streaming(orphan)}
3. 校验：该 msg 是会话最后一条 assistant，且其后无新 user 消息
4. 校验：intent=simple（knowledge/ReAct 首版不支持续传 → 409 + 提示重新提问）
5. 校验：resume_count < max-resume-attempts（可选列 resume_count 或 Redis；MVP 用 message 表列）
6. status → streaming；intent 不变
7. history = loadHistory 至该 assistant 对应的 user 消息（含 user，不含 assistant partial）
8. streamContinue(history, userMsg, partialContent) → SSE chunk（同路由）
9. 新 chunk **append** 到 DB content（非覆盖）；周期性 flush 同上
10. 终态同新消息流程
```

**SSE 事件规范（统一新消息与续传）：**

| 事件 | 示例 | 说明 |
|------|------|------|
| meta conversation | `data: {"type":"conversation","id":"<convId>"}` | 新建或确认会话 id |
| meta message | `data: {"type":"message","id":"<msgId>","status":"streaming","resume":false}` | 绑定 assistant 消息 id |
| chunk | `data: {"type":"chunk","text":"..."}` 或纯文本 chunk | MVP 可保留纯文本，meta 用 JSON |
| meta done | `data: {"type":"message","id":"<msgId>","status":"completed"}` | 终态通知 |

前端收到 `message.id` 后写入 store，刷新时以 **后端 status + content** 为准。

**LlmGatewayClient 改造：**

```java
public Flux<String> streamWithHistory(List<ChatTurn> history, String userMessage)

public Flux<String> streamContinue(List<ChatTurn> history, String userMessage, String partialAssistant)
```

- `streamContinue`：OpenAI `messages` = `system` + history + `user` + **`assistant`(content=partialAssistant，无 closing)**
- DeepSeek / OpenAI 兼容层若不支持 assistant 末条续写，fallback：`system` 注入「从以下已输出内容之后继续，不要重复：\n{partial tail 200字}」+ 常规 history（Task 实施时 Contract-first 验证）

**SunshineAgent 改造：**

```java
public Flux<String> chat(List<ChatTurn> history, String userMessage, String userId, String tenantId)
```

- history 转为 `List<Msg>`，末尾追加当前 user `Msg`
- **不再依赖** `InMemoryMemory` 存历史

**overlap trim（前端）：**

```typescript
function appendChunk(existing: string, chunk: string): string {
  const maxOverlap = Math.min(existing.length, chunk.length, 64)
  for (let n = maxOverlap; n > 0; n--) {
    if (existing.endsWith(chunk.slice(0, n))) return existing + chunk.slice(n)
  }
  return existing + chunk
}
```

## Track F — 断点续传（Orchestrator + UI）

### 问题

仅有终态/partial 持久化时，用户刷新或 Stop 后只能看到半条回复，**无法在同一轮 assistant 上继续生成**，体验断层。

### 设计要点

| 维度 | 方案 |
|------|------|
| 存储 | 同一条 `chat_message` 行追加 content；`status` 驱动 UI |
| 触发 | `POST /chat/stream` + `resumeMessageId` |
| 上下文 | 续写时不重复 append user；history 截止到该轮 user |
| 并发 | 同一条 msg 同时只允许一个 `streaming`；重复 resume → **409** |
| 范围 | MVP：**simple 直连 LLM**；knowledge/ReAct → 409，前端隐藏续传 |
| 限流 | `resume_count` 字段递增，超过 `max-resume-attempts` → 409 |

**与 Track B 周期性 flush 的配合：**

```
Client                    Orchestrator                 MySQL
  │── POST stream ────────►│ append user + assistant(streaming)
  │◄── SSE chunks ─────────│ update partial (every 500ms)
  │── disconnect ────────►│ doOnCancel → interrupted + partial
  │── refresh GET conv ───►│ read interrupted + partial
  │── POST resume ────────►│ streamContinue → append chunks
  │◄── SSE chunks ─────────│ update → completed
```

**Track F 不做（由 Track G 负责）：**

- Redis 缓冲 SSE / 客户端无感重连（见 Track G）

**Track F 仍不做：**

- 多客户端同时订阅同一条 generation 的写入冲突
- Token 级精确 checkpoint（仅 message 级 partial）

### 改动文件

- **修改** Track A/B 已有 Entity、Service、ChatController
- **修改** `LlmGatewayClient` — `streamContinue`
- **修改** `sunshine-ui` — Track D 已列

## Track G — Redis SSE 缓冲 + 客户端无感重连（阶段 1.6）

> **前置依赖**：Track A（MySQL）、Track B（流式写库 + flush）、Track C（BFF 转发）、Track D（后端数据源）。  
> **与 Track F 关系**：G 解决「**同一次 generation 仍在跑**」时的断线追读；F 解决「**generation 已停**（Stop / orphan / 服务重启）」时的 LLM 续写。生产上 **G 优先、F 兜底**。

### 问题

Track F 的「继续生成」需用户点击或重新调 LLM（`streamContinue`），且客户端断线会导致 Orchestrator 侧 Reactor 链 `cancel`、**丢失正在生成的 token**。刷新/弱网场景下体验仍不连续。

### 目标

1. LLM 生成与 SSE 推送**解耦**：客户端断开时，Job 在超时前继续消费 LLM 并缓冲 chunk  
2. 同 Tab 刷新 / 短时断网后**自动追读** Redis 中已有 chunk，无需用户点击  
3. 与 Track F 共用同一条 `chat_message`（`messageId`），MySQL partial 仍为权威终态备份  

### 架构

```
┌─────────┐   POST /chat/stream     ┌──────────────┐   独立订阅    ┌─────────────┐
│ Browser │ ───────────────────────►│ Orchestrator │◄──────────────│ LLM Gateway │
│         │◄── SSE (id=seq) ────────│              │               └─────────────┘
│         │                         │ GenerationJob│
│         │   GET reconnect         │      │       │
│         │ ───────────────────────►│      ▼       │
└─────────┘                         │ Redis Stream │
                                    │ gen:{id}     │
                                    └──────┬───────┘
                                           ▼
                                    MySQL partial flush（Track B，500ms）
```

**GenerationJob 生命周期：**

```
CREATED → RUNNING → COMPLETED
              │
              ├── client全部断开 ──► RUNNING（继续写 Redis，直到 complete 或 orphan-timeout）
              ├── orphan-timeout ──► INTERRUPTED（停 LLM，MySQL status=interrupted，可走 Track F）
              └── error ──────────► FAILED
```

### Redis 数据结构

| Key | 类型 | 内容 | TTL |
|-----|------|------|-----|
| `sunshine:gen:{generationId}` | Hash | `conversationId`, `messageId`, `userId`, `tenantId`, `status`, `lastSeq`, `intent` | `generation.ttl-sec`（默认 3600） |
| `sunshine:gen:{generationId}:events` | Stream | field `text` = chunk 正文；`seq` 与 SSE `id` 一致 | 同上 |
| `sunshine:user:{userId}:active` | String | 当前用户进行中的 `generationId`（可选，防多会话并发） | 同上 |

**Stream 写入（Job 侧）：**

```
XADD sunshine:gen:{id}:events * seq 129 text "世界"
HSET sunshine:gen:{id} lastSeq 129
```

**重连读取：**

```
XRANGE sunshine:gen:{id}:events (afterSeq + - + COUNT maxReplay
# 或 RUNNING 时用 XREAD BLOCK 订阅新条目
```

### API 设计

**1. 新开流（扩展 Track B，SSE meta 增加 generationId）**

```json
POST /api/chat/stream
{ "content": "...", "conversationId": "..." }
```

SSE 首帧 meta：

```json
{"type":"generation","id":"<generationId>","messageId":"<assistantMsgId>","seq":0}
```

后续 chunk 仍可为纯文本；SSE **`id:` 头 = seq**（单调递增整数）。

**2. 无感重连（新端点，BFF 透传）**

```
GET /api/chat/stream/{generationId}?afterSeq=128
Header: x-user-id, x-tenant-id
Accept: text/event-stream
```

- `afterSeq=0` 或省略：从第一条 chunk 重放（用于调试）  
- 校验：`generationId` 归属当前 user；`status ∈ {RUNNING, COMPLETED}`  
- 若 `COMPLETED`：重放剩余 chunk 后 emit `[DONE]`  
- 若 `RUNNING`：阻塞读 Redis Stream（`BLOCK`），新 chunk 实时转发  
- 若 `INTERRUPTED/FAILED` 且 Job 已停：**410 Gone**，前端降级走 Track F「继续生成」

**3. 可选：查询 generation 状态（非 SSE）**

```
GET /api/generations/{generationId}
→ { "status", "lastSeq", "messageId", "conversationId" }
```

供页面 mount 时判断是否自动重连。

### Orchestrator 核心组件

```
orchestrator/.../generation/
├── GenerationJob.java          # 封装 LLM Flux 消费，写 Redis + 更新 MySQL
├── GenerationRegistry.java     # 内存注册 RUNNING Job；防同 message 双 Job
├── GenerationStreamService.java # XADD / XRANGE / 订阅阻塞读
├── GenerationReconnectController.java  # GET /generations/{id}/stream
└── GenerationProperties.java   # ttl, orphan-timeout, max-buffer-chunks
```

**与 Reactor 解耦要点：**

```java
// ChatController：创建 Job 后立即返回 SSE，SSE 订阅 Redis 而非直接订阅 LLM Flux
// LLM Flux 在独立 GenerationJob 中 subscribe(onNext → XADD + append MySQL buffer)
// 最后一个 SSE 订阅者 disconnect → 不 dispose LLM 订阅（refCount / 独立 Disposable）
```

**Stop 行为：**

- 用户点 Stop → `POST /api/generations/{id}/cancel` → dispose LLM → `INTERRUPTED` → MySQL partial（与 Track F 一致）  
- 重连流收到 cancel 信号后关闭 SSE  

### BFF 改动

| 路径 | 转发 |
|------|------|
| `GET /api/chat/stream/{generationId}` | Orchestrator 同路径 |
| `GET /api/generations/{id}` | Orchestrator 同路径 |
| `POST /api/generations/{id}/cancel` | Orchestrator 同路径 |

WebClient 对 reconnect 使用 `exchangeToFlux` 透传 SSE；`afterSeq` query 原样转发。

### 前端改动（Track D 扩展）

**localStorage 轻量指针（仅当前 Tab generation）：**

```typescript
interface ActiveGeneration {
  generationId: string
  messageId: string
  conversationId: string
  lastSeq: number
}
// key: sunshine-active-generation
```

**onMounted / 切换会话：**

1. `GET /api/generations/{id}` 或读 localStorage  
2. 若 `status=RUNNING|COMPLETED` 且 `lastSeq < serverLastSeq` → **`reconnectStream(generationId, afterSeq)`**（无按钮）  
3. 若 410 / `INTERRUPTED` → 展示 partial + Track F「继续生成」  
4. 每个 chunk 更新 `lastSeq`；overlap trim 仍适用（重放边界可能与 UI 重复）

**与 Track F 优先级：**

```
刷新
 ├─ 有 active generationId 且 Job RUNNING → Track G 自动 reconnect
 ├─ Job 已 INTERRUPTED → Track F 手动 resume
 └─ 无 generation / 410 → 仅展示 MySQL partial
```
